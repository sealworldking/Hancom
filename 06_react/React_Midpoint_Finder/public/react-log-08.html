24Alert, 객체 매핑 (삼항 연산자 대신)

1) 객체 매핑이란, 종류(type)가 여러 개일 때 삼항 연산자를 길게 3중첩하지 않고, 객체를 표(사전)처럼 만들어 골라 쓸 수 있는 방법이다.
2) 핵심은 const cfg = map[type] 한 줄이다. type 값으로 map에서 그 종류의 설정 객체 전체({ icon, color })를 통째로 꺼내 cfg에 담는 코드이다.
→ 이 덕에 아래에서 cfg.color, cfg.icon처럼 점 하나로 색과 아이콘을 바로 꺼내 쓸 수 있다. 삼항 연산자를 썼을 때와는 달리, 색 따로, 아이콘 따로 구할 필요가 없는 것이다.

// src/components/Alert.jsx
const Alert = ({ type, text }) => {
  // 종류별 설정 모음 (키: {아이콘, 색})
  const map = {
    success: { icon: '👌', color: 'green' },
    error:   { icon: '🤯', color: 'crimson' },
    warning: { icon: '💀', color: 'orange' },
  };
  const cfg = map[type];   // type을 열쇠로 설정 한 번에 꺼냄
  return <p style={{ color: cfg.color }}>{cfg.icon} {text}</p>;
};
export default Alert;
  • type='success'이면 cfg는 { icon:'👌', color:'green' } 객체 전체가 된다.
  • 이렇게 설정하면, cfg.color='green', cfg.icon='👌'로 필요한 값만 꺼내 쓸 수 있다.
  • 이 방법의 장점은 if나 삼항 연산자였다면 종류가 늘었을 때 코드가 매우 길어졌을 텐데, 이 방법을 이용하면 map에 한 줄만 추가하면 되기에 효과적인 최적화가 가능하다는 것이다.

25Rating, 배열 메서드로 반복 렌더링

React로 화면을 만들다 보면 같은 요소를 개수만큼 반복해서 그려야 할 때가 많다. 이번엔 별점(Rating)을 직접 만들며, 배열 메서드로 반복 렌더링하는 법을 알아보겠다.
1) Rating 컴포넌트를 이용하면 숫자 props인 score를 받아, 별 5칸 중 앞에서부터 score개는 채운 별(⭐), 나머지는 달(🌙)로 그리게 할 수 있다.
2) 기존 방식이라면 <span>⭐</span>를 손으로 5개 일일이 적어야 하며, 별 개수가 늘면 그만큼 줄도 늘어나게 된다.
→ [...Array(5)].map을 쓰면 5칸짜리 배열을 만들어 반복하며 별 5개를 자동으로 찍는다. 직접 다 안 찍어도 되고, 가운데 숫자만 바꾸면 되므로 별이 100개여도 코드는 이 한 줄로 그대로다.

// src/components/Rating.jsx
const Rating = ({ score }) => {
  return (
    <div>
      {[...Array(5)].map((_, i) => (
        <span key={i}>{i < score ? '⭐' : '🌙'}</span>
      ))}
    </div>
  );
};
< 중요 > 스프레드 위치가 핵심
Array(5)는 값이 아예 없는 "빈 구멍" 5개라 .map이 건너뛴다.
먼저 [...Array(5)]로 펼쳐 undefined를 채워야 진짜 배열이 되어 map이 돈다.
[...Array(5).map(...)](안쪽 map)이 아니라 [...Array(5)].map(...)(바깥 map)으로 작성해야 한다.
  • ...(스프레드) = 배열을 펼치는 연산자. Array(5)의 빈 구멍을 undefined로 하나씩 채워 진짜 배열로 만든다
  • .map((_, i) => ...) = 그 배열을 돌며 각 칸을 화면 요소로 바꾼다.
    _ = 칸의 값(undefined, 안 써서 밑줄), i = 번호(0~4)
  • i < score면 별, 아니면 달 - (Ex. score=3이면 i가 0,1,2일 때만 ⭐이 찍힘)

26Tag, 배열 props + map (실무 기본형)

25번은 [...Array(5)]로 가짜 빈 배열을 만들어 반복했다. 이번엔 실제 데이터가 담긴 배열을 그대로 props로 받아 그리는 법을 알아보겠다. 상품 목록, 댓글, 검색 결과처럼 실무 데이터는 대부분 이 형태다.
1) Tag 컴포넌트는 문자열 배열(tags)을 props로 받아, map으로 각 항목을 <span>#항목</span> 형태의 태그로 그린다.
2) tags.map((tag) => ...)에서 tag는 배열의 각 요소 하나('react', 'props', ...)를 가리킨다. '#' + tag로 앞에 샵을 붙여 #react처럼 표시한다.
→ 배열에 값이 3개면 태그 3개, 10개면 10개가 자동으로 그려진다. 즉 데이터(배열)만 바꾸면 화면이 알아서 따라 바뀐다.

// src/components/Tag.jsx
const Tag = ({ tags }) => {
  return (
    <div>
      {tags.map((tag) => (
        <span key={tag}>{'#' + tag}</span>
      ))}
    </div>
  );
};

// App.jsx
const list = ['HTML', 'CSS', 'JS', 'REACT'];
<Tag tags={list} />   // 배열 통째로 넘김
key는 왜 중요한가 (값이 이름표면 중복되면?)
key는 React가 각 항목을 구분하는 고유 이름표다. 같은 값이 2개면 경고가 뜨고 목록이 바뀔 때 엉킨다.
그래서 실무에서는 데이터의 고유 id를 써서 이 문제를 해결한다 (key={t.id}).
위의 예시는 특수한 상황으로, HTML/CSS/JS/REACT가 다 다르기에 key={tag}가 안전할 뿐이다.

27MUI, 잘 만든 부품 갖다 쓰기

지금까지는 버튼, 카드 같은 부품을 직접 만들었다. 이번엔 남이 잘 만들어 둔 완성품을 가져다 쓰는 법을 알아보겠다.
1) 라이브러리란 자주 쓰는 부품, 기능을 미리 만들어 모아둔 꾸러미다. 그중 MUI(Material UI)는 구글 머티리얼 디자인 기반의 UI 부품 라이브러리로, 버튼, 입력칸, 팝업 같은 완성품을 모아 놓았다.
2) 설치(npm install) 후 import { Button } from '@mui/material' 한 줄로 부품을 가져와, variant, onClick 같은 props로 모양과 동작만 지정해 쓰면 된다.
→ 내가 CSS를 직접 짜지 않아도 디자인이 일관되고, 부품 만드는 시간이 크게 줄어든다.

// 설치: npm install @mui/material @emotion/react @emotion/styled
import { Button } from '@mui/material';

const SubmitButton = () => (
  <Button variant="contained" onClick={() => alert('안녕!')}>
    제출
  </Button>
);
  • variant, onClick 전부 props로 넘긴다. 즉, 부품은 완성돼 있고 나는 모양, 동작만 지정하면 된다.
  • variant 설정에서, variant="contained" = 꽉 찬 버튼, "outlined" = 테두리만, "text" = 글자만 선택하여, 미리 만들어진 모양 중 고를 수 있다.
  • CSS를 직접 안 짜도 디자인 일관성 확보가 가능하다는 장점이 있다. (비슷한 것: Ant Design, Chakra, shadcn/ui)
< 중요 > 서버 켠 뒤 라이브러리 설치하면 "Invalid hook call"
Vite 개발 서버가 켜진 상태에서 npm install하면 dep 캐시가 꼬여 React 인스턴스가 어긋난다.
해결: node_modules/.vite 지우고 dev 서버 재시작. 새 라이브러리 깔면 서버 한 번 껐다 켜기.

28레이아웃, 컴포넌트 위치 정하기

부품을 다 만들었으면 이제 화면에 배치해야 한다. 이번엔 여러 컴포넌트를 원하는 위치에 끼우는 법을 알아보겠다.
1) App 안에서 컴포넌트를 JSX 태그로 적으면, 적은 순서가 곧 화면에 쌓이는 순서가 된다. HTML에서 <div>를 적던 자리에 <부품 />를 적는 것과 똑같다.
2) 기본적으로는 세로로 쌓인다. 여기서, 가로로 나란히 놓으려면 부모 요소에 display:flex를 주면 된다. 순서를 바꾸려면 태그 줄을 위아래로 옮기면 된다.
→ <header>, <nav>, <main>, <footer> 같은 시맨틱 태그로 배치하면 화면은 같아도 접근성과 SEO에 유리하다.

const App = () => (
  <>
    <Header />    // 맨 위
    <Menu />      // 그 아래
    <Content />   // 또 그 아래
    <Footer />    // 맨 밑
  </>
);
  • 순서 바꾸기 = 태그 줄을 위/아래로 이동
  • 가로 배치 = 부모에 display:flex (세로 쌓기가 기본)
  • <header> <nav> <main> <footer> 같은 시맨틱 태그 = 접근성, SEO에 유리

29, 30, 31useState, 값을 기억하고 바꾸기

오늘의 메인. 지금까지 화면은 한 번 그리면 고정이었는데, 이제 값이 바뀌면 화면이 스스로 바뀌게 만든다.
→ 그 도구가 useState다.
1) useState는 값을 기억하는 상자를 만든다. props는 부모가 준 고정값이라 못 바꾸지만, state는 컴포넌트가 스스로 기억하고 바꿀 수 있는 값이다.
2) const [count, setCount] = useState(0)은 초기값 0으로 시작해, 현재값 count와 그 값을 바꾸는 함수 setCount를 돌려준다. 버튼에서 setCount로 값을 바꾸도록 설정하면, React가 화면을 자동으로 다시 그린다.
→ +1, -1, 리셋 등, 버튼이 여러 개여도 모두 같은 count를 조작한다. 즉, 어느 버튼을 눌러도 화면은 최신 count를 반영한다.

// src/components/Counter.jsx
import { useState } from 'react';

const Counter = () => {
  const [count, setCount] = useState(0);   // 값 + 바꾸는 함수 한 쌍 (초기값 0)
  return (
    <>
      <button onClick={() => setCount(c => c - 1)}>-1</button>
      <span>{count}</span>
      <button onClick={() => setCount(c => c + 1)}>+1</button>
      <button onClick={() => setCount(0)}>리셋</button>   // 초기값으로
    </>
  );
};
< 중요 > 왜 꼭 setCount로 바꿔야 하나
count = count + 1은 값만 바뀌고 React가 모른다 → 화면 그대로.
setCount(...)는 값 변경 + "화면 다시 그려" 신호를 함께 보낸다 → 화면 갱신.
c => c + 1의 c는 내가 정의하는 게 아니라 React가 넣어주는 "현재 값"이다.
  • const [count, setCount] = useState(0) = 배열 구조분해.
    → useState가 돌려주는 [현재값, 바꾸는 함수] 한 쌍을 이 두 이름으로 받는다 (이름은 내 맘대로, 관례는 값/set값)
  • c => c + 1(함수형) = 현재값 c를 받아 +1. 빠르게 여러 번 눌러도 안 꼬여서 setCount(count+1)보다 안전하다.
  • 버튼이 2개든 3개든 같은 state 하나를 조작 (증가, 감소, 초기화)
  • 리셋은 현재값이 필요 없어 setCount(0)처럼 값만 넣는다

32NameForm, 폼 입력 (controlled)

이번엔 숫자가 아니라, 사용자가 입력창에 친 글자를 state로 다루는 법을 알아보겠다.
1) 입력값을 state에 저장해 두면 검색, 로그인, 댓글처럼 사용자가 친 내용을 코드로 자유롭게 쓸 수 있다.
2) input에 value={name}으로 state 값을 보여주고, onChange={(e) => setName(e.target.value)}로 글자를 칠 때마다 state를 갱신한다. e.target.value가 지금 입력칸의 글자다.
→ state가 바뀌면 아래 <p>안녕, {name}!</p>도 즉시 따라 바뀐다. 이렇게 input과 state가 항상 일치하는 방식을 controlled(제어) 컴포넌트라 한다.

const NameForm = () => {
  const [name, setName] = useState('');
  return (
    <>
      <input value={name} onChange={(e) => setName(e.target.value)} />
      <p>안녕, {name}!</p>   // 칠 때마다 즉시 반영
    </>
  );
};
  • e.target.value = 지금 입력칸의 글자
  • value만 있고 onChange가 없으면 타이핑이 안 먹는다 (둘 다 필수)
  • {name}이 자동 갱신되는 원리: setName → 컴포넌트 함수 재실행 → 새 값으로 다시 그림

33ProductItem, Props + State 한 몸에

이번엔 지금까지 따로 배운 props와 state를 한 컴포넌트에서 같이 쓰는 법을 알아보겠다.
1) ProductItem은 상품 이름 name을 props로 받고, 담은 개수 count는 자기 state로 관리한다.
→ name은 밖(부모)에서 준 고정값이라 못 바꾸고, count는 스스로 기억하며 버튼으로 바꾼다.
2) 부모는 <ProductItem name="러닝화" />처럼 name만 넘긴다.
→ count는 부품 내부에서 useState로 알아서 관리하므로 부모는 신경 쓸 필요가 없다.
→ 같은 부품에 name만 다르게 넘기면 카드를 여러 개 만들 수 있고, 각 카드는 자기 count를 독립적으로 관리한다. 장바구니, 상품 목록이 이 구조다.

const ProductItem = ({ name }) => {        // props (밖에서 받음)
  const [count, setCount] = useState(0);   // state (내가 관리)
  return (
    <div>
      <h3>{name}</h3>
      <p>{count}개 담음</p>
      <button onClick={() => setCount(c => c + 1)}>🛒 담기</button>
    </div>
  );
};
구분props (name)state (count)
누가 주는가?부모가 줌 (밖)자기가 만듦 (안)
바꿀 수 있는가?못 바꿈 (읽기)바꿈 (setCount)
성격고정값변하는 값
  • 같은 부품에 name만 다르게 넘겨 카드 여러 개, 각 카드가 count를 독립 관리

34, 35, 36, 37useEffect, 타이밍을 잡는 도구

useState가 "값을 기억하는" 도구라면, useEffect는 "특정 타이밍에 코드를 실행하는" 도구다. 이번엔 화면이 그려진 뒤 언제 무엇을 실행할지 정하는 법을 알아보겠다.
1) useEffect(함수, 의존성배열)은 화면이 그려진 뒤 함수를 실행한다. 타이머, 서버 호출처럼 화면 밖 동작을 연결할 때 쓴다.
2) 두 번째 칸(의존성 배열)이 실행 타이밍을 결정한다.
→ []는 처음 1번, [값]은 그 값이 바뀔 때마다, 생략하면 매 렌더마다 실행된다.
→ 타이머처럼 뭔가를 켰다면 return () => ...로 정리(cleanup) 함수를 돌려줘, 컴포넌트가 사라질 때 꺼줘야 한다(아래 시계 예시 참고).

// 34) 처음 1번만
useEffect(() => { console.log('화면 뜰 때 1번'); }, []);

// 36) count 바뀔 때마다
useEffect(() => { console.log('count:', count); }, [count]);

// 37) 매 렌더마다 (거의 안 씀)
useEffect(() => { console.log('매번'); });   // 배열 생략
// 35) 시계 - setInterval + 정리(cleanup)
const Clock = () => {
  const [time, setTime] = useState(new Date().toLocaleTimeString());
  useEffect(() => {
    const timer = setInterval(() => {
      setTime(new Date().toLocaleTimeString());   // 1초마다 갱신
    }, 1000);
    return () => clearInterval(timer);   // 사라질 때 타이머 끔 (정리)
  }, []);
  return <p>🕛 {time}</p>;
};
두 번째 칸언제 실행되는가?쓰는 곳
[]처음 1번만시작 설정, 데이터 요청
[값]그 값 바뀔 때마다값 변화 감지, 자동 저장
생략매 렌더마다거의 안 씀 (무한 루프 위험)
< 중요 > useEffect가 2번 실행된다? (StrictMode)
개발 모드의 <StrictMode>는 정리(cleanup)를 제대로 했는지 검사하려고 effect를 일부러 2번 실행한다.
버그가 아니고, 실제 배포 빌드에선 1번만 돈다. StrictMode는 켜두는 게 정석.
또 setState를 배열 없는 useEffect 안에서 부르면 무한 루프가 된다(렌더→effect→setState→렌더…). 반드시 []를 넣는다.

38, 39데이터 가져오기, fetch + useEffect

오늘의 최종 목표. 실제 서버나 공개 API에서 데이터를 받아 화면에 그리는 법을 알아보겠다.
1) 흐름은 "화면이 뜨면 → 데이터를 요청하고 → 도착하면 state에 채워 화면을 갱신"이다. 요청은 useEffect(…, []) 안에서 첫 렌더 때 1번만 보낸다.
2) fetch(주소)로 요청을 보내고, .then(res => res.json())으로 응답을 JS 객체로 바꾼 뒤, .then(data => setTemp(...))로 필요한 값을 state에 담는다. 실패에 대비해 .catch도 붙인다.
→ 처음엔 빈 값(null, [])이라 "불러오는 중"을 보여주고, 데이터가 도착하면 화면이 자동으로 채워진다.

// src/components/Weather.jsx  (open-meteo 무료 날씨 API)
const Weather = () => {
  const [temp, setTemp] = useState(null);
  const [isLoading, setIsLoading] = useState(true);

  useEffect(() => {
    fetch('https://api.open-meteo.com/v1/forecast?latitude=37.5&longitude=127&current_weather=true')
      .then((res) => res.json())                // 응답 → JSON 변환
      .then((data) => {
        setTemp(data.current_weather.temperature);   // 깊이 파고들어 기온 꺼냄
        setIsLoading(false);
      })
      .catch((error) => console.error('실패:', error));
  }, []);   // [] = 첫 렌더 1번만

  return <p>🌡️ 서울 기온: {isLoading ? '불러오는 중…' : temp + '°C'}</p>;
};
  • 비동기: 데이터는 인터넷 너머라 시간차가 있다. 멈춰 기다리면 화면이 굳으니 "도착하면 그때 처리"(.then)
  • .then((res) => res.json()) = 도착한 응답(res)을 JS가 쓸 수 있는 객체로 변환. 이 변환이 끝나야 다음 .then에서 data를 쓸 수 있다
  • data.current_weather.temperature = 응답이 { current_weather: { temperature } }처럼 깊게 박혀 있어, 점(.)을 타고 상자 속 상자를 파고들어 기온만 꺼낸다
  • 주소의 ?=옵션 시작, &=옵션 이어붙이기 / 위도, 경도 바꾸면 다른 도시 (부산 35.1 / 129.0)
  • 38번(Users)과 구조 똑같음, 다른 건 주소와 값 꺼내는 경로뿐
  • 처음 빈 값(null, []) → 로딩 표시 → 도착하면 채워 갱신
async/await 버전 (같은 일, 더 읽기 쉬움)
.then 체인 대신 const res = await fetch(...)처럼 위→아래로 읽힌다.
await = "이 줄 끝날 때까지 기다렸다 다음" 이라는 뜻이다. 실무에서 많이 쓴다.

Q. setCount의 c는 정의도 안 했는데 어떻게 쓰나?

c => c + 1은 "받으면 1 더한다"는 규칙(함수)만 적은 것.
거기에 실제 현재 값을 넣어주는 건 setCount(React)의 일이다. c는 그냥 받는 자리 이름일 뿐(x, 현재값 아무거나 가능).

Q. inline style로 다 되는데 CSS는 왜 쓰나?

inline은 :hover, :active, 미디어쿼리를 못 쓰고, 재사용이 안 돼 복붙 지옥이 된다.
동적인 값(상태에 따라 바뀌는 색 등)은 inline, 재사용, hover, 반응형은 CSS. 역할이 다르다. (우선순위: inline > id > class > 태그)

Q. 화면은 흰 배경인데 글자가 회색으로 안 보였다.

템플릿 index.css의 :root가 전역 글자색을 var(--text)(회색)로 잡고, 다크모드에선 더 연해진다.
특정 요소만 바꾸려면 그 컴포넌트에서 style={{ color:'black' }}(inline이 이김), 전체를 바꾸려면 index.css의 변수 수정.

Q. 회사 이름이 왜 data.company가 아니라 data.company.name인가?

API 응답에서 company가 다시 { name, ... } 객체이기 때문. 점(.)을 타고 상자 속 상자를 한 단계 더 파고들어 company.name으로 꺼낸다. 날씨도 같은 이유로 data.current_weather.temperature.

Q. useEffect 콘솔 로그가 2번 찍힌다. 버그인가?

아니다. 개발용 <StrictMode>가 정리(cleanup) 여부를 검사하려고 일부러 2번 실행한다. 실제 배포하면 1번. 켜두는 게 정석이다.
